# Project Documentation

This document explains the files and folders that currently exist in this project folder in a beginner-friendly way.

## 1. What this project is

This is a **blockchain-based real estate ledger system**.

In simple words, it is a project that tries to store property ownership records in a way that is:

- hard to secretly change
- easy to verify
- safer against fake ownership claims
- safer against double-selling the same property

Instead of using a public blockchain like Bitcoin or Ethereum, this project builds a **small custom blockchain in Python**.

## 2. Big picture: how the project works

Think of the project as 5 layers:

1. **Frontend**
   A webpage where a user can click buttons, fill forms, and call the API.
2. **API layer**
   FastAPI endpoints receive requests like "register property" or "confirm sale".
3. **Validation layer**
   Checks if the seller is the real owner, if the property is locked, and if the signature is valid.
4. **Blockchain layer**
   Creates blocks, links them together with hashes, and checks if the chain is still valid.
5. **Database layer**
   Stores the blocks, current property ownership, and pending sale locks in SQLite.

## 3. Simple workflow example

Here is the normal sale flow in easy words:

1. A property is first registered.
2. The current owner starts a sale.
3. The system locks that property so it cannot be sold twice at the same time.
4. The seller signs the sale data using their private key.
5. The backend verifies that signature.
6. A new blockchain block is created for the sale.
7. The database updates the owner to the buyer.
8. The pending sale lock is removed.

Main endpoints:

- `GET /property/{property_id}`
- `GET /property/{property_id}/history`
- `GET /verify-owner/{property_id}`

Easy explanation:

- this file is where the app deals with creating and reading property records

Purpose:

- handles the selling process

Main endpoints:

- `POST /initiate-sale`
- `POST /confirm-sale`
- `DELETE /cancel-sale/{property_id}`

Easy explanation:

- this file manages the 2-step sale process:
  first create a pending sale, then confirm it with a digital signature

### `blockchain/block.py`

Purpose:

- defines what a block looks like

Important ideas here:

- a block stores transaction data
- a block remembers the previous block's hash
- a block computes its own SHA-256 hash

Why this matters:

- if block data changes later, the hash changes too
- that helps detect tampering

### `blockchain/chain.py`

Purpose:

- manages the list of blocks

Important methods:

- `_create_genesis_block()`
- `add_block()`
- `is_valid()`
- `get_chain()`
- `load_from_blocks()`

Beginner meaning:

- this file is the "manager" for the whole blockchain

### `blockchain/transaction.py`

Purpose:

- creates transaction objects
- creates the exact JSON payload that must be signed

Why this matters:

- both client and server must agree on the exact text being signed
- even a tiny change in order or formatting can break signature verification

### `crypto/keys.py`

Purpose:

- creates public/private key pairs

Beginner idea:

- the private key is the secret
- the public key is the identity shared with the system

### `crypto/signatures.py`

Purpose:

- signs text using the private key
- verifies signatures using the public key

Why it matters:

- this is what proves that the seller really approved the sale

### `db/schema.sql`

Purpose:

Main tables:

|---|---|

Purpose:

Important methods:

## 7. How the database and blockchain work together

This is one of the most important ideas in the project.

The project does not only use a blockchain. It uses both:

- a blockchain for **history**
- a normal database table for **current state**

That means:

- the `blocks` table keeps the full history forever
- the `properties` table keeps only the latest owner and metadata

Why this is useful:

- blockchain history is good for audit trails
- database snapshots are good for quick lookups

So if you want:

- the full story of a property, check the blockchain history
- the current owner right now, check the `properties` table

## 8. The sale flow in very simple terms

### Step 1: Register property

The system creates a registration transaction and stores it in a block.

### Step 2: Start sale

The seller says:

"I want to sell this property to this buyer for this amount."

The system then:

- checks the seller is the real owner
- checks no other sale is already in progress
- stores a pending sale lock
- returns a payload that must be signed

### Step 3: Sign payload

The seller signs that payload with their private key.

This is like a digital approval.

### Step 4: Confirm sale

The backend verifies the signature.

If valid:

- a new sale block is created
- the owner is changed to the buyer
- the pending lock is removed

## 9. Beginner glossary

### Block

A container of transaction data in the blockchain.

### Hash

A fixed-length fingerprint of data. If the data changes, the hash changes.

### Previous hash

The hash of the block before the current one. This links blocks together.

### Genesis block

The first block in the blockchain. It is created automatically.

### Public key

A shareable identity used to verify signatures.

### Private key

A secret key used to create signatures. It must never be shared.

### Signature

Proof that the owner of the private key approved some data.

### SQLite

A lightweight local database stored in one file.

### Pending sale

A temporary record that locks a property while a sale is being processed.
